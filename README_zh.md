# ecom-scene-analyzer

一个 Claude Code Skill，用于分析电商场景图（当前专精儿童学习家居），产出多维度视觉分析报告和 AI 图片生成提示词。

构建这个 Skill 的过程中，有一个体会越来越清晰：模型的能力上限远比大多数人以为的高，但这个上限不会自动浮现。它需要被唤醒——而唤醒它的方式，不是更精巧的 prompt，是你这个人看世界的方式。

你的学科视野、你的审美敏感、你区分现象和原则的能力、你对"这里可能还有一层"的直觉——这些决定了你能把 AI 带到什么深度。

从这个意义上说，Skill 不只是一个自动化工具，它是一套模型潜力的激活方案——通过精心设计的知识注入、分析视角和认知路径，把模型从停留在表象的默认状态，引导到触及事物内核的深度。

这篇文档想展开聊聊这个体会是怎么形成的。

## 视角决定深度

用过 AI 的人大概都有一个体验：同样一个问题，换一种问法，得到的回答质量可能完全不同。

这个现象很常见，但容易被当成"prompt 技巧"来理解——好像只要掌握了某种话术，就能稳定地拿到更好的结果。

构建这个 Skill 的过程让我觉得事情没那么简单。

在做场景图分析时，一开始让模型"分析一下构图、光线和人物"，得到的是很规整的基础分析。

然后换一组视角——从心理学、营销学的角度再看同一张图——分析的层次就跳了一个台阶。

模型开始讲**镜像神经元**如何让家长产生代入感，讲画面中没有文字是一种**高语境沟通**策略，讲两张图是互补的**消费者决策路径**。

这些知识模型一直都有。但在第一轮对话里，它们全部沉默着。

它们需要一个信号才会被调用——而这个信号就是你给它的分析视角。

---

这件事在后续的迭代中反复得到验证。

加入**文化符号学**的视角，模型能读出一幅地面上的儿童画框是"这个家庭重视艺术教育"的阶层认同信号。

加入**感官联觉**的视角，它能分析出窗纱暖光触发了温度联想，让平面图片产生了物理空间感。

每加一个维度，不是多出一段话的问题，而是打开了一个之前完全不可见的分析层面。

所以回到一开始那个现象——"换一种问法得到不同质量的回答"——它的本质不是话术差异，是**视角差异**。

模型的分析能力更像一个函数：输入是你提供的视角集合，输出是对应深度的分析。视角越丰富、越跨学科，输出就越有层次。

---

这个理解直接影响了 Skill 的设计方向。

一个 Skill 的核心工作，不是约束模型的行为——"你必须按这个格式输出"——而是**为它配置一组观察世界的透镜**。

这个 Skill 里的"归因工具箱"——六个分析视角：视觉设计原理、心理学机制、营销策略、品类特异性、文化符号学、感官联觉——做的就是这件事。

但紧跟着就有一个不太舒服的问题：**谁来决定放哪些透镜？**

选择"该从心理学角度看"，前提是你自己对消费者心理有基本的感知。选择"该从感官联觉角度看"，前提是你站在这张图面前时，自己能感受到暖光带来的温度联想。

如果你完全没有这些感知，你就不会想到把这些视角放进工具箱——不是因为不知道这些概念存在，而是缺乏那个直觉，把一个跨学科的概念和眼前这张具体的商业图片联系起来。

这种直觉很难用"技能"来描述。它更像是一种长期浸泡在不同学科、不同审美体验中之后，慢慢沉淀下来的敏感度。

一种对"这里可能还有一层"的嗅觉。

这意味着 Skill 设计者的认知边界，会成为 Skill 产出质量的天花板。Skill 可以被复制，但决定 Skill 里放什么透镜的那个判断力，复制不了。

## 观察不是原则

构建过程中踩了一个很有意思的坑。

最初几轮对话都是基于几张具体的图片。模型在分析中归纳出了一些视觉特征——木地板、浅色长绒地毯、孩子不看镜头、暖黄色调——然后在抽象成 Skill 的时候，这些特征直接被写成了规则。

好像所有好的儿童学习椅场景图都得有木地板和浅色地毯。

拿新的图去验证就暴露了问题。一张以灰色瓷砖和白墙为背景、孩子正面微笑的图，各方面都很优秀，但按 Skill 里的"规则"来看全是违规项。

---

问题出在哪里呢。

木地板、浅色地毯、不看镜头——这些是**观察**，不是**原则**。

它们是那几张特定图片中的具体选择，不是"一张好的场景图应该是什么样"的普遍规律。

真正的原则藏在观察的下面一层：地面材质应该与整体空间风格统一，传递目标客群认同的家居审美。这条原则在不同场景里会表现为完全不同的地面材质，都合理。

这个错误其实很普遍——不止 AI 会犯，人也经常犯。我们天然倾向于从有限的样本中提取模式，然后把模式当成规律。

看到三张好图都用了暖色调，就觉得"好图应该用暖色调"。但暖色调只是那三张图中"传递温馨安全感"这个原则的一种表现形式。冷色调在另一个语境下也可以传递安全感，只要整体视觉语言是统一的。

---

意识到这一点之后，Skill 的架构做了一次比较大的调整。

把编码在 Skill 里的东西分成了三层：

- **硬约束**——物理法则（人椅比例、透视一致性、光影方向），不可违反
- **品类默认值**——行业共识（暖色调、柔光、整洁环境），可以被覆盖
- **用户自定义**——Case 级别的选择（具体服装颜色、房间类型、道具），完全灵活

Skill 应该编码原则，不应该编码现象。

这听起来像一句正确的废话，但在实操中区分两者没有看起来那么容易。

---

## 归因才是灵魂

Skill 能跑起来之后，遇到了另一个问题。

产出的分析很详细、维度很全，但读起来总觉得隔了一层。

后来想清楚了：它在做识别，没有在做归因。

---

"主光从右侧 45° 打入，柔光，低反差"——这是识别。

"选择柔光是因为硬阴影投射在儿童身上会在潜意识中触发'不安全'联想，而母婴品类的核心情绪需求是安全感"——这是归因。

两者的差距不是详略问题，是**性质问题**。

识别告诉你"是什么"。归因告诉你"为什么"。

---

识别是不可复用的——"这张图用了柔光"这个事实对下一张图毫无帮助。

但"儿童品类的柔光本质上是在服务安全感需求"这条归因是可以复用的，它可以被带到任何一个与安全感相关的品类中去。做包装设计、做母婴店铺的空间灯光、做儿童 App 的界面色调，这条归因都成立。

更重要的是，归因才能让读到分析的人产生**认同感**。

识别让人觉得"嗯，对"，归因让人觉得"原来如此"。

一份只有识别的分析报告，读完之后你知道了一些事实但不太确定为什么要知道它们。一份有归因的报告，读完之后你理解了一张图背后的整套说服逻辑，而且你知道这套逻辑可以怎么迁移。

---

这个 Skill 最终设置了六个归因视角，不是作为一个"越多越好"的清单，而是为了覆盖"人为什么会被一张图说服"这个问题的不同层面。

它可能在视觉设计层面有效，也可能在心理学层面有效，也可能在文化认同层面有效。这些层面不互斥，同一个视觉元素经常同时在多个层面上起作用。

在归因之上还有一层，叫**元思考**。

归因回答"这张图为什么有效"，元思考回答"这背后有什么可以带到其他领域去的通用原理"。

比如从"侧面是展示坐姿的最佳角度"归因出"因为它像一张坐姿X光片"，再跳到"所有以人体状态改善为核心卖点的产品，拍摄角度都应优先考虑状态可读性而非产品外观"。

后者就是元思考——一条可以迁移到护腰靠枕、人体工学键盘等完全不同品类的认知框架。

这是分析和洞察的分界线。分析在具体 Case 的范围内有效，洞察跨 Case 有效。

Skill 如果只要求归因，产出的是分析；要求元思考，才有可能产出洞察。

回过头看这几层设计——视角注入、归因工具箱、元思考要求——它们做的其实是同一件事：把模型从"描述表象"逐步推向"抓住内核"。模型不缺抓内核的能力，缺的是一个信号告诉它"不要停在表面，继续往下走"。Skill 的每一层设计都是这个信号。

## 从分析到生成

这个 Skill 的另一半是提示词生成——基于分析结果，产出可以直接用于 AI 图片生成工具的提示词。

> **关于生成模型的选择**：生成的提示词建议在 Gemini 系列模型中使用，目前在 Nano Banana 2（Gemini 3.1 Flash Image）上实测效果不错。提示词的效果与生成模型的智力水平直接相关——模型越强，对提示词中视觉语义、空间关系、气质描述的理解越准确，产出质量差距会比较明显。

这里提两个设计选择。

**第一个是"第零步"。** 在填任何参数之前，先用一句话描述一个完整的生活画面——"安静的午后，一个乖巧的女孩在书桌前写作业，窗外阳光透过薄纱照进来"。

有了这个画面，服装颜色、道具选择、光线方向就不再是需要逐项决策的孤立参数，它们从画面中自然长出来。

这是为了解决模型"从参数表出发拼装"导致画面缺乏生命力的问题。先整体后局部，而不是先局部后拼装。

**第二个是 content 和 design_notes 的分离。** 提示词的正文（content）是给 AI 生成工具的纯执行指令，设计笔记（design_notes）是给设计师的决策解释。

模型在写提示词时会不自觉地把"为什么选这个颜色"混进"画面里有什么颜色"——它有一种解释自己选择的倾向。把两者分开，是基于观察到的具体认知倾向而设计的结构性对策。

类似的小设计还有不少——比如要求变体必须覆盖不同维度（而非换措辞）、比如自检是否包含画幅比例和主体占比——每一条背后都是迭代中遇到的真实问题。

累积起来，它们构成了一个对模型认知倾向的理解图谱。

## 让思考被看到

结果的呈现也值得花功夫。

分析内容被渲染成一份交互式 HTML 报告：原图与总体印象并排展示，五维分析、核心洞察、风格参数、提示词四个 Tab 切换，滚动时 Tab 栏置顶、原图缩小悬浮供对照。

不同认知层次的内容用不同的视觉层级承载——观察和归因分开呈现，核心洞察的三层递进（归因→元思考→复用建议）各有独立区块。

## 回到开头

这个 Skill 的结构——三个知识文件的分离、五维分析框架、六个归因视角、认知阶梯——这些都是可以复制的。

换一个品类，替换领域知识文件，框架依然成立。

但构建这个框架时那些判断——在工具箱里放感官联觉而不是别的什么视角，要求元思考而不是停在归因层，把观察和原则区分开而不是混为一谈——这些判断来自设计者自身的积累。

不是某一种特定的积累，而是所有阅读、体验和思考沉淀下来的那种对事物的敏感度。

---

模型的能力是摆在那里的，对每个人都一样。但每个人能把它唤醒到什么程度，不一样。

差异不在于谁掌握了更好的 prompt 技巧，而在于谁能给它更丰富的视角、更准确的判断、更深层的追问。

这些东西的来源，是你在心理学中读到的对人性动机的理解，是你在美术馆里培养的超越语言的感知力，是你在哲学训练中形成的质疑前提的习惯，是你所有经历沉淀出的对世界的敏感度。

AI 是已知世界的总和。人文素养深厚的人，是这个总和的指挥家——不需要演奏每一种乐器，但知道在什么时刻该让什么声部响起来。

这个 Skill 是一次尝试。它试图把一段对话中涌现的认知路径冻结下来，让更多人可以复用。

但它不只是一个可复用的工具。使用这个 Skill 的过程，本身就是一次模型潜力的激发实践——你会看到品类知识如何为模型补充垂直领域的认知盲区，归因工具箱如何把它从"描述现象"推向"解释内核"，认知阶梯如何引导它从分析跃迁到洞察。这套从表象到内核的激发逻辑，不止适用于电商场景图分析，它是一种可以迁移到任何领域的 Skill 设计方法论。

而这也是最终的提醒：真正决定产出深度的，不是 Skill 的结构，而是设计 Skill 的那个人，是否有能力抓住事物的内核而非停留在表象——以及愿意在自身的厚度上投入多少。

还有一个更远的问题值得想。模型的智慧水平在快速提升，但人类的认知带宽和知识储备始终存在上限。当模型的能力天花板越来越高，而人类作为指挥家的带宽没有同比增长时，如何更有效地激发模型潜力，可能会成为人与 AI 协作中最核心的命题。这个 Skill 是对这个命题的一次小规模实践，但远不是答案。答案需要更多人一起探索。

## 快速开始

**前置条件**：[Claude Code](https://docs.anthropic.com/en/docs/claude-code) CLI 或桌面端，Python 3.6+

**安装**：从 [Releases](../../releases) 下载 `.skill` 文件，在 Claude Code 中安装。

**使用**：上传电商场景图，说"帮我分析这张场景图"。Skill 自动触发，产出交互式 HTML 分析报告。

---

## 品类适配

1. 替换 `references/category-knowledge.md`——写入你的品类知识。关键：写原则不写现象。不是"用柔光"，而是"柔光服务于安全感需求——硬阴影触发不安全联想"。
2. 基本保留 `references/analysis-guide.md`——五维框架和归因工具箱是品类无关的。
3. 按需调整 `references/prompt-guide.md` 和 `SKILL.md`。
